# SQL 代码规范

本文档定义了 MaxCompute SQL 代码的编写规范，**每次生成 SQL 前必须完整阅读**。

---

## 一、生成原则（最高优先级）

### 1.1 多口径指标必须询问（强制执行，最高优先级）

**当指标说明中存在多个同名或别名相同的指标时，必须停止并询问用户选择口径。**

**执行流程**：
1. 在指标说明中搜索指标（检查指标名称和指标别名列）
2. 统计匹配数量：
   - 匹配到 1 个 → 继续生成 SQL
   - 匹配到多个 → **立即停止，列出所有口径，询问用户选择**

**询问格式**：
```
找到多个指标口径，请选择：

1. [指标名称] - [口径说明]
2. [指标名称] - [口径说明]
...

请回复数字选择。
```

**绝对禁止**：
- ❌ 匹配到多个指标时直接生成 SQL
- ❌ 自行选择其中一个口径

### 1.2 字段命名规则（硬性限制）

**所有字段名、表别名、CTE 名称必须使用英文，禁止使用中文**。

**优先级顺序**：
1. 数据字典中已定义的英文字段名（最高优先级）
2. 下划线分隔的英文命名
3. 业务语义明确的英文缩写

**正确示例**：
- 字段别名：`AS fiscal_year`、`AS sold_amt`
- 表别名：`t_order`、`t_member`
- CTE 名称：`order_agg`、`daily_stats`

**禁止示例**：
- `AS 财年`、`AS 售卖金额` → 必须使用英文
- CTE `售卖数据` → 必须使用英文命名

**强制检查**：生成 SQL 后，逐个检查所有字段别名、表别名、CTE 名称，确保无中文。

### 1.3 样例 SQL 优先（强制执行）

- **强制规则**：严格遵循指标说明中样例 SQL 的表选择、关联方式、指标聚合方式
- **⚠️ 样例SQL中的警告必须执行**：
  - 需要UNION多张表 → 必须使用UNION ALL合并，禁止只查询一张表
  - 需要去重 → 必须按指定字段去重
  - 需要特殊处理 → 必须执行特殊处理逻辑
- 样例与通用规则冲突时，优先遵循样例

**检查方法**：生成SQL后，对照指标说明检查是否执行了所有警告。

### 1.4 字段存在性规则

- **强制规则**：所有字段引用必须在数据字典中存在，禁止使用不存在的字段
- **检查方法**：生成 SQL 后，逐个检查每个字段是否在对应表的字段说明中
- **UNION 规则**：两表字段数量、类型必须一致，用 NULL 填充缺失字段
- 优先使用字段说明中的现成字段/枚举值，避免复杂计算

### 1.5 聚合粒度规则（重要）

**核心原则**：COUNT(DISTINCT) 类指标必须在最终聚合粒度上进行去重统计。

**正确做法**：
```sql
-- 正确：先聚合到目标粒度，再COUNT(DISTINCT)
SELECT
    f_year,
    f_week,
    COUNT(DISTINCT member_id) AS member_cnt
FROM fact_table f
LEFT JOIN dim_calendar d ON f.date = d.dateid
GROUP BY f_year, f_week
```

**错误做法**：
```sql
-- 错误：先在细粒度COUNT(DISTINCT)，再SUM累加
-- 这会导致同一会员在不同日期被重复统计
SELECT
    f_year,
    f_week,
    SUM(member_cnt)  -- 错误！
FROM (
    SELECT
        date,
        COUNT(DISTINCT member_id) AS member_cnt
    FROM fact_table
    GROUP BY date
) t
GROUP BY f_year, f_week
```

**判断方法**：
- 如果指标含义是"XX数"（会员数、订单数等）→ 必须在最终粒度COUNT(DISTINCT)
- 如果指标含义是"XX金额"、"XX数量" → 可以先在细粒度聚合，再SUM

### 1.6 CTE 设计原则（重要）

**核心规则**：
- **尽可能减少 CTE 数量**，能合并的尽量合并
- 每个 CTE 对应一个**业务主题**，完成该主题的全部处理
- CTE 内部完成所有聚合，**禁止在最终查询中再次聚合或关联明细表**

**CTE 结构模式**：

```sql
-- 推荐结构：按业务主题划分
WITH
  -- CTE 1: 维度关联（如日历维度）
  dim_cte AS (
    SELECT dimension_fields
    FROM dimension_table
    WHERE conditions
  ),

  -- CTE 2: 业务主题A（如售卖数据）
  -- 包含：关联维度 + 聚合指标 + 维度字段
  theme_a AS (
    SELECT
      d.dimension_fields,  -- 维度字段
      SUM(metric1) AS metric1,  -- 聚合指标
      COUNT(DISTINCT metric2) AS metric2
    FROM fact_table_a f
    LEFT JOIN dim_cte d ON f.key = d.key
    WHERE f.conditions
    GROUP BY d.dimension_fields
  ),

  -- CTE 3: 业务主题B（如核销数据）
  -- 包含：关联维度 + 聚合指标 + 维度字段（与主题A相同的维度）
  theme_b AS (
    SELECT
      d.dimension_fields,  -- 与主题A相同的维度字段
      SUM(metric3) AS metric3,
      COUNT(DISTINCT metric4) AS metric4
    FROM fact_table_b f
    LEFT JOIN dim_cte d ON f.key = d.key
    WHERE f.conditions
    GROUP BY d.dimension_fields
  )

-- 最终查询：仅做主题间的关联，不再聚合
SELECT
  a.dimension_fields,
  a.metric1,
  a.metric2,
  b.metric3,
  b.metric4
FROM theme_a a
LEFT JOIN theme_b b ON a.dimension_fields = b.dimension_fields
```

**设计要点**：
1. **维度先行**：先定义维度CTE，后续业务CTE关联使用
2. **主题独立**：每个业务主题的CTE独立完成维度关联和指标聚合
3. **维度对齐**：所有业务主题使用相同的维度字段进行GROUP BY
4. **最终简单**：最终查询只做主题间的LEFT JOIN，不再聚合

**绝对禁止**：
- ❌ 最终查询中再次 GROUP BY 或聚合
- ❌ 最终查询中关联明细表或中间结果表
- ❌ 业务主题CTE中未完成所有指标的计算

**精简原则**：
- **能合并的 CTE 尽量合并**，减少 CTE 数量
- 同一数据源的不同处理步骤必须合并为一个 CTE
- 简单聚合的 CTE 优先内联到主查询
- 使用 CASE WHEN 替代多个条件统计 CTE

**精简示例**：

```sql
-- 错误：过度拆分（8 个 CTE）
WITH step1 AS (...), step2 AS (...), step3 AS (...), step4 AS (...),
     step5 AS (...), step6 AS (...), step7 AS (...), step8 AS (...)

-- 正确：按业务主题合并（3 个 CTE）
WITH
  dim_cte AS (...),           -- 维度关联
  theme_a AS (...),           -- 业务主题A（已完成聚合）
  theme_b AS (...)            -- 业务主题B（已完成聚合）
SELECT ... FROM theme_a LEFT JOIN theme_b ON ...
```

## 二、CTE 规范（重要）

### 2.1 核心原则

- **尽可能减少 CTE 数量**，能合并的尽量合并
- **禁止行为**：同一数据源的多个处理步骤拆成多个 CTE
- **合并优先级**：同一表 → 同一粒度 → 同一业务主题

### 2.2 必须合并的场景

| 场景 | 错误做法 | 正确做法 |
|------|----------|----------|
| 同一表的多次聚合 | 每个聚合一个 CTE | 合并为一个 CTE，一次 GROUP BY |
| 多数据源 UNION | UNION 前后各一个 CTE | 子查询内 UNION，外层聚合 |
| 关联维度表 | 关联前后各一个 CTE | 在同一 CTE 内完成关联 |
| 去重后聚合 | 去重一个 CTE，聚合一个 CTE | 子查询去重，外层聚合 |
| 条件统计 | 每个条件一个 CTE | CASE WHEN 条件聚合 |
| 会员统计单独CTE | 会员明细一个CTE，最终再JOIN | 在业务主题CTE中用CASE WHEN完成统计 |

### 2.3 条件聚合技巧

```sql
-- 使用 CASE WHEN 合并不同条件的统计
SELECT
    <维度>,
    COUNT(DISTINCT CASE WHEN <条件1> THEN <字段> END) AS <指标1>,
    COUNT(DISTINCT CASE WHEN <条件2> THEN <字段> END) AS <指标2>
FROM table 
GROUP BY <维度>
```

### 2.4 CTE 检查清单

生成 SQL 后，逐项检查：

- [ ] 是否已尽可能精简 CTE 数量？
- [ ] 同一数据源是否合并为一个 CTE？
- [ ] 每个 CTE 是否完成了所有聚合？
- [ ] 最终查询是否只做主题间关联，不再聚合？
- [ ] 最终查询是否未关联明细表？

## 三、聚合规则

### 3.1 核心原则：样例 SQL 优先

**聚合方式必须以指标说明中的样例 SQL 为准，禁止自行推断**。

| 指标样例 SQL 中 | 处理方式 |
|----------------|----------|
| 直接 SUM | 直接聚合，不做去重 |
| 有子查询 + MAX | 粒度提升，按指定字段去重 |
| COUNT(DISTINCT) | 去重计数 |

**禁止行为**：
- 禁止对样例 SQL 中直接 SUM 的字段做粒度提升
- 禁止将不同字段的聚合方式混用
- 禁止忽略样例 SQL 中的警告标注

### 3.2 聚合类型对照表

| 类型 | 判断依据 | 处理方式 |
|------|----------|----------|
| 直接聚合 | 样例 SQL 中直接 SUM | 直接 SUM/COUNT |
| 去重聚合 | 样例 SQL 中 COUNT(DISTINCT) | COUNT(DISTINCT) |
| 粒度提升聚合 | 样例 SQL 中有子查询 + MAX | 按样例中的字段去重 |

### 3.3 聚合检查清单

**必须先查看指标说明中的样例 SQL**，然后检查：
- [ ] 样例 SQL 中该字段是直接聚合还是粒度提升？
- [ ] 如果是粒度提升，按哪个字段去重？
- [ ] 是否有警告标注需要特殊处理？

## 四、语法规范

### 4.1 遵守 MaxCompute 官方语法

- 所有函数和语法必须符合阿里云 MaxCompute 官方文档
- 禁止使用其他数据库（MySQL、Hive、PostgreSQL）特有的函数或语法
- 不确定时，查阅 [MaxCompute 官方文档](https://help.aliyun.com/document_detail/27825.html)

**常见差异**：

| 功能 | MaxCompute 支持方式 |
|------|---------------------|
| 字符串拼接 | CONCAT() 函数，不支持 `||` 操作符 |
| 日期格式化 | DATE_FORMAT() 或 FROM_UNIXTIME() |
| 条件表达式 | CASE WHEN 或 IF() 函数 |
| 类型转换 | CAST() 或隐式转换 |

### 4.2 禁止 SELECT 嵌套

所有复杂查询必须使用 CTE（WITH 语句）组织，禁止在 SELECT 中嵌套子查询。

```sql
-- 错误：SELECT 嵌套
SELECT * FROM (
    SELECT * FROM (
        SELECT * FROM table_a
    ) t1
) t2

-- 正确：使用 CTE
WITH base AS (
    SELECT * FROM table_a
)
SELECT * FROM base
```

## 五、性能规范

### 5.1 必须筛选分区

读取事实表时**必须**增加分区筛选条件，避免全表扫描。

```sql
-- 错误：无分区筛选
SELECT * FROM sbux_cdm.dwd_pmt_disct_redemption_line_di

-- 正确：筛选分区
SELECT * FROM sbux_cdm.dwd_pmt_disct_redemption_line_di
WHERE ds = '20260320'
```

**常用分区字段**：
- `ds`：日期分区，格式为 `yyyymmdd`

### 5.2 避免不必要的排序

如无特殊要求（如取 Top N），**不需要**添加 `ORDER BY` 子句。

**何时需要 ORDER BY**：
- 用户明确要求排序输出
- 需要取 Top N 的场景
- 窗口函数中需要排序

### 5.3 JOIN 优化

- 减少关联次数，能一次 JOIN 完成的不要拆分
- 小表在前，大表在后

## 六、代码风格

### 6.1 保持精简

- 移除未使用的字段和 CTE
- 统一别名风格，避免无意义命名

### 6.2 缩进规范

```sql
-- 推荐格式
WITH base AS (
    SELECT
        order_id,
        member_id,
        pay_amt
    FROM sbux_cdm.dwd_pmt_disct_redemption_line_di
    WHERE ds = '20260320'
),
agg AS (
    SELECT
        member_id,
        COUNT(DISTINCT order_id) AS order_cnt,
        SUM(pay_amt) AS total_amt
    FROM base
    GROUP BY member_id
)
SELECT * FROM agg
```

### 6.3 别名规范

- 表别名：使用有意义的英文简写（如 `t_order`、`t_member`）
- 字段别名：与业务含义一致的英文命名
- CTE 名称：体现业务含义的英文命名（如 `order_agg`、`daily_stats`）
- **禁止使用中文作为字段名、表别名、CTE 名称**

## 七、字段命名规范

### 7.1 核心原则（硬性限制）

**禁止使用中文字段名**：所有字段名、表别名、CTE 名称必须使用英文。

**优先级**：
1. 使用数据字典中已定义的英文字段名
2. 下划线分隔的英文命名
3. 业务语义明确的英文命名

### 7.2 检查规则

- 字段名必须反映真实的业务含义
- 禁止将 A 含义的字段命名为 B 含义的名称
- 优先使用数据字典中定义的英文字段名
- 聚合字段可在原名后加后缀

### 7.3 检查方法

逐个检查 SELECT 子句中的输出字段：
1. 是否为英文命名？
2. 是否与业务含义一致？
3. 是否使用了数据字典中的字段名？

## 八、检查清单（必须执行）

生成 SQL 后，逐项检查：

### 指标计算逻辑校验（最重要，必须执行）
- [ ] 对照指标说明，检查每个指标的计算逻辑是否符合要求
- [ ] 需要粒度提升的指标：是否按指定字段MAX后再SUM？
- [ ] 需要去重的指标：是否使用了COUNT(DISTINCT)？
- [ ] 需要UNION多表的指标：是否包含了所有表的数据？
- [ ] **检查方法**：逐个指标对照样例SQL，确认聚合方式一致

### 字段命名检查（必须执行）
- [ ] **所有字段名使用英文（禁止中文）**
- [ ] **所有表别名使用英文（禁止中文）**
- [ ] **所有 CTE 名称使用英文（禁止中文）**
- [ ] 字段名与业务含义一致
- [ ] 优先使用数据字典中的英文字段名

### 字段引用检查（重要）
- [ ] 每个字段引用是否存在于其数据源（表或 CTE）
- [ ] CTE 中引用的字段是否在上游 CTE 的 SELECT 中输出
- [ ] GROUP BY 后是否遗漏了需要传递的字段

**检查方法**：从最内层 CTE 开始，逐层向上检查每个 SELECT 引用的字段是否可在上游找到。

### 样例SQL警告检查（重要）
- [ ] 对照指标说明，检查所有警告是否已执行
- [ ] 需要UNION多张表的警告是否已执行

### CTE 检查（重要）
- [ ] 列出所有 CTE 名称：________________（用逗号分隔）
- [ ] 是否已尽可能精简 CTE 数量？
- [ ] 同一数据源是否合并为一个 CTE
- [ ] 每个 CTE 是否完成了该业务主题的全部处理
- [ ] 最终查询是否只做主题间关联，不再聚合
- [ ] 最终查询是否未关联明细表

### 聚合粒度检查（重要）
- [ ] COUNT(DISTINCT)类指标是否在最终聚合粒度上进行
- [ ] 是否避免了"先COUNT再SUM"的错误做法

### 语法检查
- [ ] 所有函数符合 MaxCompute 官方语法

### 性能检查

## 九、常见问题

| 问题 | 正确做法 |
|------|----------|
| 忽略样例SQL警告 | 对照指标说明检查所有警告是否已执行，如需要UNION多张表必须执行 |
| 使用不存在的字段 | 检查数据字典，确保所有字段在对应表的字段说明中存在 |
| UNION 字段不匹配 | 确保两表字段数量、类型一致，用 NULL 填充缺失字段 |
| 使用中文字段名或别名 | 改用英文命名，优先使用数据字典中的英文字段名 |
| CTE 数量过多 | 合并同一数据源的 CTE，内联简单聚合，使用 CASE WHEN 替代多个条件统计 CTE |
| 同一数据源拆成多个 CTE | 合并为一个 CTE，用子查询或 CASE WHEN 处理不同逻辑 |
| 最终查询中再次聚合 | 在业务主题CTE中完成所有聚合，最终查询只做关联 |
| 最终查询关联明细表 | 在业务主题CTE中完成所有计算，禁止最终查询关联明细表 |
| COUNT(DISTINCT)在细粒度聚合 | 必须在最终聚合粒度上进行COUNT(DISTINCT)，禁止先COUNT再SUM |
| CTE 引用了上游未输出的字段 | 检查上游 SELECT 是否包含该字段，或补充到 GROUP BY |
| 多个 CTE 聚合同一张表 | 合并为一个 CTE，一次 GROUP BY |
| 使用 `LIMIT` 无排序 | 加上 `ORDER BY` 明确排序规则，或移除 `LIMIT` |
| 多层嵌套子查询 | 改用 CTE 逐层定义 |
| 忘记分区筛选 | 检查所有事实表，添加 `WHERE ds = ...` |
| 字段过多难以阅读 | 每行一个字段，对齐格式 |
